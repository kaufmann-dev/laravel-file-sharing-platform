Warum Laravel?

Grundfunktionen:

- Bundles oder kleine Funktionspakete
- Automatisches Laden der Klasse
- Unit-Tests
- Composer anzeigen
- Anwendungslogik
- Ruhige Controller
- Umgekehrtes Routing

Vorteile:

1. **nutzt neuesten Funktionen von PHP**

Framework nutzt alle neuen Funktionen von PHP. (Namespaces, Interfaces, Overloading, Anonymous Functions und Shorter Array Syntax)

1. **Großartige Dokumentation**

Alle Versionen von Laravel werden mit entsprechender Dokumentation veröffentlicht, gute und detaillierte Erklärungen zu Codierungsstil, Methoden und Klassen.

1. **Integration mit Mail-Diensten**

Es gibt Treiber für SMTP, Mailgun, SparkPost, Mandrill, die „Mail“-Funktion von PHP, Amazon SES und „sendmail“, um beim Versenden von E-Mails über Cloud-basierte oder lokale Dienste zu helfen.

4.**Unterstützt beliebte Cache-Backends**

Laravel unterstützt Cache-Backends wie Memcached und Redis out-of-the-box.

1. **Hat ein integriertes Tool für die Befehlszeile namens Artisan**

Das Kommandozeilen-Tool namens Artisan hilft beim Erstellen eines Skelettcodes und verwaltet das Datenbanksystem erfolgreich. Es hilft bei der Generierung der grundlegenden MVC-Dateien und der Verwaltung von Assets, einschließlich ihrer jeweiligen Konfigurationen.

1. **Pakete und Ressourcenverfügbarkeit**

Von npm-Paketen und Bower-Paketen profitieren, indem man das Framework mit Gulp und elixir kombiniert. Diese Integration mit Composer löst Abhängigkeiten und ist somit die zuverlässigste Paketquelle in der PHP-Welt.

1. **Reduzierter Produktentwicklungszyklus**
2. **IoC-Container**

Auch Inversion of Control genannt, ist es eine Methode zum Generieren neuer Objekte und man muss keine externen Bibliotheken booten.

1. **Umgekehrtes Routing**

Dies ist eine sehr nützliche Funktion im Framework, mit der man Links zu benannten Routen erstellen kann. Wenn man also die Links erstellen, verwendet man einfach den Namen des Routers, und das Framework fügt automatisch die richtige URL ein. Und wenn man in Zukunft die Routen ändern müssen, werden die Änderungen automatisch überall vorgenommen.

1. **Beredtes ORM**

Es bietet eine einfache und sehr schöne ActiveRecord-Implementierung für die Arbeit mit Ihrer Datenbank. Modelle die man im MVC erstellt haben eine entsprechende Tabelle in der Datenbank. Das ORM verfügt über eingebaute Beziehungen. Wenn man also eine Tabelle in der Datenbank bearbeitet, verwaltet man auch die zugehörigen Daten. Die folgenden Beziehungen sind möglich => eins-zu-eins, eins-zu-n, n-zu-n, hat-n-durch und n-zu-n polymorphe Beziehungen.

## Laravel Quickstart:

## Via Laravel-Installer

Über das Laravel-Installationsprogramm
zunächst das Laravel-Installationsprogramm mit Composer herunterladen

```php
**composer global require "laravel/installer=~1.1"**
```

Sicherstellen, das ~/.composer/vendor/bin directory im richtigen Pfad (bei Windows) → C:\%HOMEPATH%\AppData\Roaming\Composer\vendor\bin
Dies ist wichtig, dass Laravel die Datei findet wenn man es ausführt

Nach der Installation wird mit dem einfachen Befehl *laravel new* eine neue Laravel-Installation in dem von Ihnen angegebenen Verzeichnis erstellt. So würde beispielsweise *laravel new blog* ein Verzeichnis namens blog erstellen, das eine neue Laravel-Installation mit allen installierten Abhängigkeiten enthält. Diese Installationsmethode ist wesentlich schneller als die Installation über den Composer.

## Via Composer

Das Laravel-Framework verwendet Composer für die Installation und die Verwaltung von Abhängigkeiten. Nun kann man Laravel installieren, indem man folgenden Befehl ins Terminal eingibt:

```php
**composer create-project laravel/laravel your-project-name 4.2.***
```

Mit diesem Befehl wird eine neue Kopie von Laravel heruntergeladen und in einem neuen Ordner des Projektes im aktuellen Verzeichnis installiert.

Wenn man es bevorzugen, kann man alternativ eine Kopie des Laravel-Repositorys von GitHub manuell herunterladen. Als nächstes führt man den Befehl *composer install* im Stammverzeichnis des manuell erstellten Projekts aus. Dieser Befehl wird die Abhängigkeiten des Frameworks herunterladen und installieren.

Berechtigungen
Nach der Installation von Laravel muss man dem Webserver eventuell Schreibrechte für die Verzeichnisse *app/storage* erteilen. Weitere Einzelheiten zur Konfiguration findet mna in der Installationsdokumentation.

Bedienung von Laravel
Normalerweise kann man einen Webserver wie Apache oder Nginx verwenden, um die Laravel-Anwendungen zu betreiben. Wenn man mit PHP 5.4+ arbeitet und den in PHP integrierten Entwicklungsserver verwenden möchtet, kann man den Befehl serve Artisan verwenden:

```php
**php artisan serve**
```

Standardmäßig ist der HTTP-Server auf Port 8000. Wenn dieser Port jedoch bereits in Gebrauch ist oder man mehrere Anwendungen auf diese Weise bedienen möchte, kann man dies beim zu verwendenden Port angeben. 

```php
**php artisan serve --port=8080**
```

### Beispielprojekt (mvc)

Das Model
Erstellung eines Modells, das ein Auto darstellen soll.

Laravel verfügt über eine fantastische, integrierte Befehlszeilenschnittstelle, die Artisan CLI, die eine Reihe nützlicher Befehle zur Verfügung stellt, die bei der Erstellung der Anwendung unterstützen.

In der Kommandozeile, ins Hauptverzeichnis der Anwendung wechseln und führen den folgenden Befehl ausführen, um ein neues Automodell zu erstellen:

```php
$ php artisan make:model Car --migration
```

Die Modelle werden alle im Hauptverzeichnis der Anwendung gespeichert, so dass dieser Befehl eine Modelldatei app/Car.php mit dem folgenden Code erzeugt:

```php
<?php

namespace App;

use Illuminate\Database\Eloquent\Model;

class Car extends Model
{
    //
}
```

Aufgrund der in Laravel integrierten Modellfunktionalität geht Laravel bei der Erstellung einer leeren Modellklasse davon aus, dass dieses Modell mit einer Datenbanktabelle namens cars verbunden ist.

Und tatsächlich hat Laravel durch die Angabe der Option --migration bei der Erstellung des Modells auch eine Datenbankmigrationsdatei für die Erstellung der Datenbanktabelle cars erstellt. 
Die Migrationsdatei befindet sich unter [timestamp]_create_cars_table.php und enthält den folgenden Code:

```
<?php

use Illuminate\Database\Schema\Blueprint;
use Illuminate\Database\Migrations\Migration;

class CreateCarsTable extends Migration
{
    public function up()
    {
        Schema::create('cars', function (Blueprint $table) {
            $table->increments('id');
            $table->timestamps();
        });
    }

    public function down()
    {
        Schema::drop('cars');
    }
}

	...

?>
```

Jetzt muss man nur noch die Dokumentation des Schema-Builders von Laravel verwenden, um die Migrationsdatei zu vervollständigen. So kann man einige zusätzliche Spalten definieren, um beispielsweise die Marke, das Modell und das Produktionsdatum der Autos zu speichern:

```php
 Schema::create('cars', function (Blueprint $table) {
    $table->increments('id');
    $table->string('make');
    $table->string('model');
    $table->date('produced_on');
    $table->timestamps();
  });
```

Anschließend kann man die Migration ausführen, um die Cars-Tabelle mit dem folgenden Artisan-Befehl zu erstellen:

```php
$ php artisan migrate
```

Wie in der vorangegangenen Tabelle wird die Seite Auto anzeigen unter http://app.url/cars/{car} aufgerufen. 
In diesem Fall handelt es sich bei {car} um die ID eines Autoobjekts in der Datenbank.

Die URL für die Anzeige des Fahrzeugs mit der Kennung 1 wäre also beispielsweise http://app.url/cars/1.

Um die Seite Show Car zu implementieren, muss man in der Show-Aktion des Controllers Folgendes tun

1) Das Car-Modell verwenden, um das gewünschte Car-Objekt aus der Datenbank abzurufen.
2) Einen View für die Seite Show Car laden und ihm das aus der Datenbank abgerufene Car-Objekt übergeben.
Um auf das Car-Modell im Controller zugreifen zu können, müssen wir zunächst eine neue Use-Anweisung oberhalb der Controller-Klasse

```
use App\Car;

class CarController extends Controller
{
	.......
}
```

Dann können wir die Show-Aktion mit folgendem Code erweitern:

```
 public function show($id)
    {
      $car = Car::find($id);
      return view('cars.show', array('car' => $car));
    }
```

Wenn man die URL von Auto 1 aufruft (http://app.url/cars/1) macht Laravel die erste in der URL über die Variable $id in der oben gezeigten show-Funktion zugänglich.

Um das Auto-Objekt mit Hilfe des Car-Modells abzurufen, muss lediglich Car::find aufgerufen und die $id übergeben werden.

Die Ansicht wird dann mit der Funktion view geladen, wobei der Name der Ansicht (die wir in einer Minute erstellen werden) und ein Array mit den Daten, die an die Ansicht geliefert werden sollen, übergeben werden.

Zuletzt muss man noch die Show-Ansicht erstellen.

View + Fertigstellung der Seite

Die Laravel-View-Dateien werden alle im Ordner resources/views gespeichert. 
Innerhalb dieses Verzeichnisses können sie in Unterordnern organisiert werden.

Im vorherigen Schritt wurde der View-Funktion den View-Namen cars.show übergeben. Dadurch wird Laravel angewiesen, nach einer View-Datei zu suchen, die sich in einem Unterordner namens cars (innerhalb des Hauptverzeichnisses resources/views) mit dem Namen show.blade.php befindet.

Laravel-Ansichtsdateien verwenden die Blade-Templating-Engine und werden daher .blade.php genannt.

Um die Implementierung der Seite Show Car abzuschließen, kann man die Ansichtsdatei resources/views/cars/show.blade.php mit dem folgenden Code erstellen:

```php
<!DOCTYPE html>
	<html>
		<head>
			<title>Car {{ $car->id }}</title>
		</head>
		<body>
			<h1>Car {{ $car->id }}</h1>
				<ul>
					<li>Make: {{ $car->make }}</li>
					<li>Model: {{ $car->model }}</li>
					<li>Produced on: {{ $car->produced_on }}</li>
				</ul>
		</body>
</html>
```

Da man das Car-Objekt in der show-Aktion des Controllers mit dem Array-Schlüssel car an die View übergeben hat, kann man in der View über eine gleichnamige Variable, $car, darauf zugreifen.

(Objekte, die über ein Modell abgerufen werden, sind Instanzen dieser Modellklasse.)

### Views und Routes:

Was ist eine Route?
Eine route in Laravel ist im Grunde ein Endpunkt, der durch eine URI spezifiziert wird und als "Zeiger" auf eine von der Anwendung angebotene Funktionalität fungiert. 
In den meisten Fällen verweist eine Route einfach auf eine Methode eines Controllers und legt fest, welche HTTP-Methoden diesen URI erreichen können. 

Eine Route bedeutet auch nicht immer eine Controllermethode; sie kann auch einfach die Ausführung der Anwendung an eine definierte Closure oder anonyme Funktion übergeben.

Warum eine Route verwenden?

Routen werden in Dateien unter dem Ordner /routes im Stammverzeichnis des Projekts gespeichert. Standardmäßig gibt es einige verschiedene Dateien, die den verschiedenen "Seiten" der Anwendung entsprechen.

- web.php
    - öffentlich zugänglichen "Browser"-basierten Routen
    - werden vom Webbrowser aufgerufen
    - laufen über die Web-Middleware-Gruppe
    - enthalten auch Funktionen für den CSRF-Schutz
    - enthalten im Allgemeinen einen gewissen "Zustand"
    
- api.php
    
    API-Middleware standardmäßig aktiviert
    
    - zustandslos
    - haben keine Sitzungen
    - haben keinen anfrageübergreifenden Speicher

- console.php
    - entsprechen den benutzerdefinierten artisan-Befehlen, die man für die Anwendung erstellt hat
- channels.php
    - Registriert Routen für die Übertragung von Ereignissen
- /upload
    - URI der Hauptseite, die unser Webformular zum Hochladen von Dateien anzeigt
- /process
    
    über das Formular (/upload-URI) übermittelten Daten veröffentlichen
    
- /list
    
    alle auf die Website hochgeladenen Dateien aufgelistet

## Laravel Datenbanktransaktionen

Eine Datenbanktransaktion ist eine Reihe von Vorgängen, die man innerhalb der Datenbankstruktur einer Anwendung durchführen kann.

Beispielcode um schnell und einfach accounts und user zu erstellen.

Sieh dir diesen Beispielcode an:

```php
// Create Account
$newAcct = Account::create([
  'accountname' => Input::get('accountname'),
]);

// Create User
$newUser = User::create([
  'username' => Input::get('username'),
  'account_id' => $newAcct->id,
]);
```

Hier gibt es zwei Szenarien, die unangenehme Probleme verursachen können:

1. Konto wird nicht erstellt
2. Fehler bei der Erstellung eines Benutzers

**Migrationen**

- eine wichtige Funktion in Laravel
    - Tabellen erstellen/bearbeiten
    - Änderungen vorzunehmen
    - Datenbankschema zu teilen
    - Spalten hinzfügen/entfernen

### **Seeders**

- Testen verschiedener Datentypen zu erleichtern
- Fehler zu beheben
- Leistung optimieren
- automatisch mehrere Zeilen mit Dummy-Daten befüllen

## **Optionen für Laravel-Datenbanktransaktionen**

- **Erstelle eine Transaktion:**

```php
DB::beginTransaction();
```

- **Eine Transaktion rückgängig machen**

```php
 DB::rollBack();
```

- **Bestätige eine Transaktion:**

```php
DB::commit();.
```

—>  jede offene Transaktion mit einer Commit- oder Rollback-Aktion abzuschließen

Andernfalls wird diese manuelle Methode nicht mehr synchronisiert und Datensätze werden nicht aktualisiert. 

# Laravel Livewire

Tools wie Vue und React sind extrem leistungsfähig, aber die Komplexität, die sie dem Workflow eines Full-Stack-Entwicklers hinzufügen, ist wahnsinnig.

Livewire ist ein Full-Stack-Framework für Laravel, das die Erstellung dynamischer Schnittstellen vereinfacht, ohne dass man den Komfort von Laravel verlassen muss.

```php
use Livewire\Component;
```

Laravel Livewire ist eine library , die es einfach macht, 

- moderne
- reaktive
- dynamische

Schnittstellen mit Laravel Blade als Templating-Sprache zu erstellen. 

Bei der Verwendung von Livewire werden die Routen Ihrer Anwendung mit typischen Blade-Vorlagen reagieren. Innerhalb dieser Templates können Sie jedoch nach Bedarf Livewire-Komponenten rendern:

```php
<div class="mt-4">
    @livewire('server-list')
</div>
```

Components
Bei der Entwicklung des Jetstream Livewire-Stacks wurde eine Reihe von Blade-Komponenten (Schaltflächen, Panels, Eingaben, Modals) entwickelt, um die Konsistenz der Benutzeroberfläche und die Benutzerfreundlichkeit zu verbessern. Es steht Ihnen frei, diese Komponenten zu verwenden oder nicht zu verwenden. Wenn Sie sie jedoch verwenden möchten, sollten Sie sie mit dem Artisan-Befehl vendor:publish veröffentlichen:

php artisan vendor:publish --tag=jetstream-views
Sie können einen Einblick in die Verwendung dieser Komponenten erhalten, indem Sie ihre Verwendung in den vorhandenen Jetstream-Ansichten im Verzeichnis resources/views Ihrer Anwendung überprüfen.

#Modals

Die meisten Komponenten des Jetstream Livewire-Stacks kommunizieren nicht mit Ihrem Backend. Die in Jetstream enthaltenen modalen Livewire-Komponenten interagieren jedoch mit Ihrem Livewire-Backend, um ihren offenen/geschlossenen Zustand zu bestimmen.

Darüber hinaus enthält Jetstream zwei Arten von Modals: Dialog-Modals und Bestätigungs-Modals. Das Bestätigungsmodal kann verwendet werden, um destruktive Aktionen wie Löschungen zu bestätigen, während das Dialogmodal ein allgemeineres modales Fenster ist, das jederzeit verwendet werden kann.

Bsp.: Account löschen

```php
<x-jet-confirmation-modal wire:model="confirmingUserDeletion"><x-slot name="title">
        Delete Account
    </x-slot><x-slot name="content">
        Are you sure you want to delete your account? Once your account is deleted, all of its resources and data will be permanently deleted.
    </x-slot><x-slot name="footer"><x-jet-secondary-button wire:click="$toggle('confirmingUserDeletion')" wire:loading.attr="disabled">
            Nevermind
        </x-jet-secondary-button><x-jet-danger-button class="ml-2" wire:click="deleteUser" wire:loading.attr="disabled">
            Delete Account
        </x-jet-danger-button></x-slot></x-jet-confirmation-modal>
```

Wie man sieht, ist wird der modal's open / close state  durch die wire:model property bestimmt, die in der Komponente deklariert ist.

Der Name der Eigenschaft sollte einer booleschen Eigenschaft in der entsprechenden PHP-Klasse Ihrer Livewire-Komponente entsprechen. 

Normalerweise setzt man diese Eigenschaft auf true, wenn der Benutzer auf ein UI-Element in der Anwendung klickt, das das Modal öffnen soll. Die Eigenschaft sollten auf false gesetzt werden, wenn man bereit ist, das Modal zu schließen.

### Beispiel:

```php

use Tests\TestCase;

use Illuminate\Support\Facades\Route;
use Illuminate\Foundation\Testing\RefreshDatabase;

class ProfileTest extends TestCase
{
    use RefreshDatabase;

    function profile_page()
```

## Tips und Tricks

1) Kein render() erforderlich

Wenn die render()-Methode jedoch nur eine einzeilige Methode ist, um die Standardansicht zu rendern, kann man diese aus der Komponente löschen, und alles wird weiterhin funktionieren. Das ist der Grund, da die Standardmethode render() von der Methode des Anbieters (vendor’s) geladen wird.

Anstatt:

```php
class PostsShow extends Component
{
	public function render()
	{
		return view('livewire.posts-show');
	}
}
```

Kann man es einfach so schreiben:

```php
class PostsShow extends Component
{
}
```

Es macht so gesehen genau das gleiche und beide Varianten funktionieren

2) Komponenten in Unterordner

Wenn man eine Komponente in einem Unterordner generieren will, wie z.B. app/Http/Livewire/Ordner/Component.php, hat man zwei Möglichkeiten, dies zu tun:

php artisan make:livewire Folder/Component
oder

php artisan make:livewire folder.component
Bei der ersten Variante ist der erste Buchstabe großgeschrieben wird, bei der zweiten klein geschrieben.

Unterordner werden automatisch erstellt, wenn sie noch nicht existieren.

3) Komponente in non-default Folder

Komponenten in nicht standardmäßigen Ordnern
Wenn man ein externes Paket mit Livewire-Komponenten verwenden, kann es sein, dass sich Livewire-Komponente in einem anderen Ordner als dem Standardordner app/Http/Livewire befinden. In diesem Fall muss man den Namen der Komponente an den tatsächlichen Speicherort binden.

Normalerweise geschieht dies in der Methode boot() in app/Providers/AppServiceProvider.php :

```php
class AppServiceProvider extends ServiceProvider
{
	public function boot()
	{
		Livewire::component('shopping-cart', \Modules\Shop\Http\Livewire\Cart::class);
	}
}
```

4) Einfaches Umbenennen oder Verschieben von Komponenten
Wenn man sich bei der Erstellung der Komponente mit make:livewire vertippt hat, kann man die beiden Dateien nicht manuell umbenennen, dafür gibt es einen Befehl.

Wenn man z.B. php artisan make:livewire Prduct geschrieben hat, aber stattdessen "Product" wollte, und außerdem beschlossen hat, die Komponente in einen Unterordner zu verschieben, kann das mit diesen Befehl gemacht werden:

```php
php artisan livewire:move Prduct Products/Show

```

Das Ergebnis wird so aussehen:

```php
COMPONENT MOVED
CLASS: app/Http/Livewire/Prduct.php
=> app/Http/Livewire/Products/Show.php
VIEW:  resources/views/livewire/prduct.blade.php
=> resources/views/livewire/products/show.blade.php
```

## Für’s Projekt:

[https://laravel-livewire.com/docs/2.x/file-uploads](https://laravel-livewire.com/docs/2.x/file-uploads)

****File Uploads****

1) WithFileUploads-Eigenschaft zu Komponenten hinzufügen

2) Nun kann man wire:model auf Dateieingaben anwenden

Beispiel hochladen von Fotos:

```php
use Livewire\WithFileUploads; 

class UploadPhoto extends Component{
	use WithFileUploads;
	public $photo; 

	public function save(){
		$this->validate(['photo' => 'image|max:1024', // 1MB Max    
		]);
		$this->photo->store('photos');
	}
}
```

```php
<form wire:submit.prevent="save">
    <input type="file" wire:model="photo">

    @error('photo') <span class="error">{{ $message }}</span> @enderror

    <button type="submit">Save Photo</button>
</form>

```

Wenn eine neue Datei ausgewählt wird, sendet das JavaScript von Livewire eine erste Anfrage an die Komponente auf dem Server, um eine temporäre „signierte“ Upload-URL zu erhalten.

Sobald die URL empfangen wurde, führt JavaScript den eigentlichen „Upload“ auf die signierte URL durch, speichert den Upload in einem von Livewire festgelegten temporären Verzeichnis und gibt die eindeutige Hash-ID der neuen temporären Datei zurück.

Sobald die Datei hochgeladen und die eindeutige Hash-ID generiert wurde, sendet das JavaScript von Livewire eine letzte Anfrage an die Komponente auf dem Server, um ihr mitzuteilen, dass die gewünschte öffentliche Eigenschaft auf die neue temporäre Datei gesetzt werden soll.

Jetzt ist die öffentliche Eigenschaft (in diesem Fall $photo) auf den temporären Dateiupload eingestellt und kann jederzeit gespeichert oder validiert werden.

**Storing Uploaded Files**

Das vorherige Beispiel zeigt das einfachste Speicherszenario: 
Verschieben der vorübergehend hochgeladenen Datei in das Verzeichnis "photos" auf der Standardfestplatte des Dateisystems der App.

Möglicherweise möchtet man jedoch den Dateinamen der gespeicherten Datei anpassen oder sogar ein bestimmtes Speicher-"Laufwerk" angeben, auf dem die Datei gespeichert werden soll (z. B. in einem S3-Bucket).

Livewire verwendet dieselben APIs wie Laravel, um hochgeladene Dateien zu speichern. Hier sind jedoch ein paar gängige Speicherszenarien:

```php
// Store the uploaded file in the "photos" directory of the default filesystem disk.
$this->photo->store('photos');
// Store in the "photos" directory in a configured "s3" bucket.
$this->photo->store('photos', 's3');

// Store in the "photos" directory with the filename "avatar.png".
$this->photo->storeAs('photos', 'avatar');

// Store in the "photos" directory in a configured "s3" bucket with the filename "avatar.png".
$this->photo->storeAs('photos', 'avatar', 's3');

// Store in the "photos" directory, with "public" visibility in a configured "s3" bucket.
$this->photo->storePublicly('photos', 's3');

// Store in the "photos" directory, with the name "avatar.png", with "public" visibility in a configured "s3" bucket.
$this->photo->storePubliclyAs('photos', 'avatar', 's3');
```

**Handling Multiple Files**

Livewire verarbeitet automatisch mehrere Datei-Uploads, indem es das Mehrfach-Attribut im <input>-Tag erkennt.

Hier ist ein Beispiel für einen Datei-Upload, der mehrere Uploads verarbeitet:

```php
use Livewire\WithFileUploads; 

class UploadPhotos extends Component{ 
	use WithFileUploads; 
	public $photos = [];

	public function save(){
	$this->validate(['photos.*' => 'image|max:1024', // 1MB Max13        
	]);
	foreach ($this->photos as $photo){
	 $photo->store('photos');
		}
	}
}
```

```php
<form wire:submit.prevent="save">
    <input type="file" wire:model="photos" multiple>

    @error('photos.*') <span class="error">{{ $message }}</span> @enderror

    <button type="submit">Save Photo</button>
</form>

```
